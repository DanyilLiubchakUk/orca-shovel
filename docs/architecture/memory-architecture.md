# Memory Architecture

## Memory Layers

1. Raw sources in Convex.
2. Extracted memory facts in Convex.
3. Profiles in Convex.
4. Namespace tags such as `@home`, `@work`, and `@kitchen`.
5. Semantic chunks in Qdrant.
6. Audit and AI run logs in Convex.

Qdrant improves retrieval. It is not the truth store.

## Memory Pipeline

```text
raw source created
  -> Trigger.dev processing task
  -> classify source
  -> chunk text
  -> extract candidate facts
  -> sensitivity/risk classification
  -> write candidate or active facts to Convex
  -> generate embeddings
  -> upsert Qdrant points with user-scoped payload
  -> update profiles when useful
  -> log AI run and memory events
```

## Candidate vs Active Rules

Auto-active may be allowed for low-risk factual items:

- work project names
- non-sensitive preferences
- simple tasks
- obvious upcoming events

Candidate review should be required for:

- health
- finance
- legal
- identity documents
- private relationship details
- emotional/mental health
- location patterns
- sensitive workplace conflicts
- facts with low confidence

## Retrieval Modes

### Factual Answering

Use:

- semantic Qdrant search
- metadata filters
- source records
- memory facts

Do not include writing style unless the user asks for a draft in their voice.

### Personalized Drafting

Use:

- recent context profile
- style profile
- relevant memory facts
- relationship/project context
- source snippets when needed

### Automation Planning

Use:

- user request
- current capabilities
- policy rules
- existing automation policies
- recent context only when relevant

## Required Memory Package API

`packages/memory` should expose functions like:

- `searchMemory({ tenantId, userId, query, filters, timeWindow, limit })`
- `upsertMemoryVector({ tenantId, userId, point })`
- `deleteMemoryVector({ tenantId, userId, pointId })`
- `rebuildUserMemoryIndex({ tenantId, userId })`
- `chunkSourceText({ source })`
- `rankMemoryResults({ taskType, results })`
- `linkRawSourceToVectors({ tenantId, userId, sourceRef, pointIds })`
- `deleteVectorsForRawSource({ tenantId, userId, sourceRef })`

It should not export raw Qdrant clients to app code.

## Qdrant Filter Contract

Qdrant filters are generated by one central builder. Required filters are hard constraints, not optional caller preferences:

- `tenantId`
- `userId`
- active status
- enabled visibility
- time window when the task implies one
- sensitivity/category limits when policy requires them
- namespace tags when the user asks for `/home`, `/work`, or similar scoped retrieval

Optional filters can narrow by source type, memory type, project, person, tag, confidence, or date range. Optional filters never replace the required filters.

## Namespace Tags vs Profiles

Profiles are compact system summaries used for personalization. Namespace tags are user-managed folders/scopes.

Examples:

- `@home`: user-managed memory/source grouping.
- `/home`: chat retrieval instruction to use the `@home` namespace.
- style profile: system summary of writing style.
- recent context profile: rolling summary controlled by the user's `recentContextDays` setting.

The two systems can reference each other, but they are not the same thing.

## Expiring Memories

Temporary memories can have `expiresAt`. Before expiry, the app should:

- show a "soon expiring" section in the dashboard
- send periodic email digests if enabled
- ask whether the memory is still relevant
- let the user delete, keep, or extend

Repeated extensions should extend farther each time, for example one month, then two months, then longer, using constants from the config package.

## Raw and Vector Linkage

Raw sources and Qdrant points must be linked both ways:

- raw source stores derived memory/vector references
- vector payload stores source IDs and memory IDs

Deleting a raw source requires confirmation explaining that linked memories and vectors will be deleted. Once confirmed, delete from Convex source tables, memory facts where applicable, Qdrant vectors, and audit the deletion.

## Qdrant Decision

Qdrant is selected over Upstash Vector for this project because semantic retrieval is a core product layer. The app needs strong metadata filtering, user isolation, payload search, future hybrid retrieval, and long-term retrieval quality control.
